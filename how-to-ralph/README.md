# Ralph Loop

Implementation of the Ralph Wiggum Loop based on Geoffrey Huntley's method and Theo (t3.gg)'s experience.

## What is Ralph?

Ralph is an LLM orchestration method that enables autonomous software development in a loop.

**Core principle**: Each iteration starts with a fresh context, avoiding "context rot" that makes models progressively dumber.

> "Think like a C or C++ engineer. There's no memory on the server side. It's a sliding window over the array. You want to set only one goal and objective in that array."
> — Geoffrey Huntley

## Why Ralph?

Traditional LLM conversations accumulate context (tokens) over time. This causes:

1. **Context rot** - The model becomes less effective as the conversation grows
2. **Snowball effect** - Errors compound, the model forgets objectives
3. **Compaction loss** - Auto-summarization loses critical information

Ralph solves this by **resetting context completely** after each task. The LLM's memory lives in files on disk, not in the conversation.

## File Structure

```
ralph-loop/
├── ralph.sh                # Main loop script
├── prompt.md               # Instructions injected each iteration
├── specs.md                # The what/why (generated by LLM)
├── implementation_plan.md  # Phases + tasks + notes (generated by LLM)
├── decisions.md            # Frozen choices (prevents flip-flops)
├── status.md               # Stop condition
├── kickoff_specs.md        # Template to generate specs
├── kickoff_plan.md         # Template to generate implementation plan
├── CLAUDE.md               # Instructions for the agent (auto-read by Claude Code)
└── README.md               # Documentation
```

### File Descriptions

| File | Purpose | Who writes it |
|------|---------|---------------|
| `prompt.md` | Instructions for each iteration | You (once) |
| `specs.md` | Context, goals, constraints, out of scope | LLM + You review |
| `implementation_plan.md` | Phases, tasks, status, notes | LLM (updated each iteration) |
| `decisions.md` | Architectural choices made | LLM (append only) |
| `status.md` | Stop condition (`status: done`) | LLM (when finished) |

## How ralph.sh Works

### The Loop Logic

1. **Initialization** - Parse arguments to determine mode (auto/manual) and max iterations. Load `prompt.md` as instructions and `status.md` as stop condition.

2. **Iteration start** - Check if max iterations reached. If yes, exit.

3. **Execute Claude** - Pipe `prompt.md` content to Claude Code in headless mode (`-p`). In auto mode, runs with streaming output and verbose logging. In manual mode, cleaner output without streaming/verbose for easier supervision. Both modes auto-approve tool calls. Claude reads the prompt, executes one task, updates files, then exits.

4. **Check completion** - After Claude exits, check if `status.md` contains `status: done`. If yes, the loop ends.

5. **Loop** - Increment counter, wait 2 seconds, restart from step 2 with a completely fresh context.

### Modes

| Mode | Description |
|------|-------------|
| **auto** (default) | Full automation with streaming output. Uses `--dangerously-skip-permissions`, `--output-format=stream-json`, `--verbose` |
| **manual** | Cleaner output for manual supervision. Uses `--dangerously-skip-permissions` only (no streaming/verbose) |

### Key Flags Explained

| Flag | Purpose | Mode |
|------|---------|------|
| `-p` | Print mode (non-interactive, exits after completion) | Both |
| `--dangerously-skip-permissions` | Auto-approve all tool calls (YOLO mode) | Both |
| `--output-format=stream-json` | Real-time streaming output | Auto only |
| `--verbose` | Detailed execution logging | Auto only |

### Stop Conditions

The loop exits when:
1. `status.md` contains `status: done` (LLM signals completion)
2. Max iterations reached (if specified)
3. Manual `Ctrl+C`

### Usage

```bash
./ralph.sh             # Auto mode, unlimited iterations
./ralph.sh 20          # Auto mode, max 20 iterations
./ralph.sh manual      # Manual mode (Ctrl+C to stop)
```

## Complete Workflow

```
┌─────────────────────────────────────────────────────────────────┐
│  PHASE 1: Preparation (conversation with LLM)                   │
│                                                                 │
│  1. Describe your need to the LLM                               │
│  2. LLM generates specs.md                                      │
│  3. You review/edit                                             │
│  4. LLM generates implementation_plan.md                        │
│  5. You review/edit                                             │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  PHASE 2: Execution (autonomous Ralph Loop)                     │
│                                                                 │
│  ./ralph.sh runs in a loop:                                │
│    1. Read specs.md + implementation_plan.md + decisions.md     │
│    2. Pick the most important task                              │
│    3. Execute it                                                │
│    4. Update implementation_plan.md                             │
│    5. If decision made → add to decisions.md                    │
│    6. Run tests                                                 │
│    7. If tests pass → commit                                    │
│    8. If ALL done → write "status: done" in status.md           │
│    9. Context reset, loop restarts                              │
└─────────────────────────────────────────────────────────────────┘
```

**You = architect/reviewer**
**LLM = writer + executor**

## File Formats

### specs.md

```markdown
# Specs: My Project

## Context
[Why this project exists]

## Goal
[What we want to achieve]

## Constraints
[Technical limitations, time, dependencies]

## Hypotheses
[Assumptions we're making]

## Unknowns
[What we don't know yet]

## Out of scope
[What we're NOT doing]
```

### implementation_plan.md

```markdown
# Implementation Plan

## Phase 1: Setup
**Status:** ✅ DONE
- [x] Init project
- [x] Config dependencies

**Notes:** Project created with Vite.

---

## Phase 2: Core Features
**Status:** 🔄 IN_PROGRESS
- [x] Feature A
- [ ] Feature B

**Notes:** Feature A done. Next: Feature B.

---

## Phase 3: Tests
**Status:** ⏳ PENDING
- [ ] Unit tests
- [ ] E2E tests

**Notes:**
```

Status values: `⏳ PENDING` | `🔄 IN_PROGRESS` | `✅ DONE`

The agent updates this file after each iteration. It's the **persistent memory**.

### decisions.md

```markdown
# Decisions

- **PostgreSQL** — Need native JSONB. Rejected: MySQL, SQLite.
- **REST** — Sufficient for scope. Rejected: GraphQL.
- **Raw SQL** — Critical perfs. Rejected: ORM.
```

Format: `**What** — Why. Rejected: alternatives.`

The agent **reads** this file before each decision and **appends** new choices. It can never modify or delete an existing decision.

### status.md

```markdown
status: in_progress

## Notes
<!-- High-level notes between iterations -->
```

When all tasks are complete, the agent writes `status: done`.

## Quick Start

### 1. Generate specs

```bash
claude "I want to create [your project].
        Generate a specs.md with context, goals, constraints."
```

Review and edit `specs.md` if needed.

### 2. Generate implementation plan

```bash
claude "Based on specs.md, generate an implementation_plan.md
        with clear phases and tasks."
```

Review and edit `implementation_plan.md` if needed.

### 3. Launch the loop

```bash
chmod +x ralph.sh
./ralph.sh
```

### 4. Supervise

The loop runs until the agent writes `status: done` in `status.md`.

You can:
- Watch iterations in real-time
- Edit files between iterations if needed
- Stop with `Ctrl+C` if things go wrong

## Security

`--dangerously-skip-permissions` gives full access to your machine.

**Recommendations:**
- Use in an isolated VM or container
- No unnecessary network access
- No credentials/secrets in the environment
- Set max iterations as a safety net

## Key Concepts

### Human ON the loop, not IN the loop

You supervise without blocking each action. The LLM works autonomously, you review results.

### Backpressure

Create backpressure via tests, typechecks, lints, builds that reject invalid work. The prompt says "run tests" generically, your project config makes it specific.

### One context window, one goal

Each iteration has one objective. Don't try to do everything at once.

> "One context window, one activity, one goal. That goal can be very fine-grained like do a refactor, add structured logging."
> — Geoffrey Huntley

### Files are memory

The LLM remembers nothing between iterations. Everything important must be written to disk:
- Progress → `implementation_plan.md`
- Decisions → `decisions.md`
- Completion → `status.md`

## Comparison: Plugin vs Bash Loop

| Aspect | Anthropic Plugin | Bash Loop (this) |
|--------|-----------------|------------------|
| Compaction | Auto (lossy) | Never (full reset) |
| Determinism | No | Yes |
| Control | Less | Total |
| Context rot | Yes | No |

We use the bash loop approach for maximum control and determinism.
